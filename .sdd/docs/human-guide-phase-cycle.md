# Работа с SDD: полный цикл одной фазы

Практическое руководство для человека-оператора. Описывает всё, что **ты делаешь руками**,
и всё, что делает LLM — чтобы было понятно, где чья зона ответственности.

---

## Что такое фаза

Фаза — единица работы в SDD. Каждая фаза имеет:

- **Спецификацию** (`Spec_vN`) — формальные требования, инварианты, границы изменений
- **План** (`Plan_vN`) — декомпозиция спецификации на milestone'ы
- **TaskSet** (`TaskSet_vN`) — список задач с точными Input/Output путями
- **State** (`State_index.yaml`) — текущее состояние (проекция, никогда не редактируется вручную)

Цикл фазы: `PLANNED → ACTIVE → COMPLETE`

---

## Обзор: кто что делает

| Шаг | Кто | Действие |
|-----|-----|---------|
| 1. Draft Spec | LLM | Пишет черновик спецификации в `specs_draft/` |
| 2. Approve Spec | **Человек** | Проверяет черновик, перемещает в `specs/` |
| 3. Plan Phase | LLM | Создаёт `Plan_vN.md` на основе утверждённой спецификации |
| 4. Decompose | LLM | Разбивает план на задачи в `TaskSet_vN.md` |
| 5. Activate Phase | **Человек** | Запускает `sdd activate-phase N --tasks T` |
| 6. Implement + Validate | LLM | Реализует и проверяет каждую задачу по очереди |
| 7. Supervise | **Человек** | Ревьюит PR/коммиты после каждой задачи или серии задач |
| 8. Check DoD | LLM | Проверяет Definition of Done (`sdd check-dod`) |
| 9. Summarize + Snapshot | LLM | Пишет отчёт и снапшот EventLog |
| 10. Approve Completion | **Человек** | Даёт финальное добро на закрытие фазы |

---

## Пример: фаза 17 от начала до конца

Допустим, ты начинаешь новую фазу N=17.

---

### Шаг 1 — LLM пишет черновик спецификации

Ты говоришь LLM:

```
Draft Spec_v17
```

LLM создаёт `.sdd/specs_draft/Spec_v17_*.md` по шаблону из `.sdd/templates/Spec_template.md`.

**Ты не делаешь ничего** — ждёшь файл.

---

### Шаг 2 — Ты проверяешь и утверждаешь спецификацию

Читаешь `.sdd/specs_draft/Spec_v17_*.md`. Проверяешь:

- Цель фазы понятна и ограничена
- Инварианты (I-XXX) формулированы однозначно
- Нет выхода за рамки домена проекта
- Нет противоречий с предыдущими спецификациями

Если всё устраивает — **перемещаешь файл**:

```bash
cp .sdd/specs_draft/Spec_v17_*.md .sdd/specs/
```

> Это единственный способ утвердить спецификацию. LLM не имеет права писать в `.sdd/specs/`.

Если нужны правки — говоришь LLM что исправить, он переписывает черновик. Повторяешь до готовности.

---

### Шаг 3 — LLM создаёт план

```
Plan Phase 17
```

LLM читает утверждённую спецификацию через `sdd show-spec --phase 17` и создаёт
`.sdd/plans/Plan_v17.md` со статусом `PLANNED`.

---

### Шаг 4 — LLM декомпозирует план на задачи

```
Decompose Phase 17
```

LLM создаёт `.sdd/tasks/TaskSet_v17.md`. Каждая задача содержит:

- `Inputs:` — точные пути к читаемым файлам
- `Outputs:` — точные пути к изменяемым файлам
- `Invariants:` — какие инварианты проверяет
- `Acceptance:` — критерий приёмки

Ты просматриваешь TaskSet. Убеждаешься:

- Задачи независимы и атомарны (одна задача = один Outputs-файл или связанная группа)
- Нет задач с расплывчатыми Inputs/Outputs (`*`, glob-паттерны)
- Количество задач разумное (10–30)

---

### Шаг 5 — Ты активируешь фазу

**Это твоё действие. LLM не может его выполнить.**

```bash
sdd activate-phase 17 --tasks 15
```

Где `--tasks 15` — количество задач в TaskSet (считаешь строки `T-17xx` в TaskSet).

Команда эмитирует два события атомарно:
- `PhaseStartedEvent` — фаза 17 становится ACTIVE
- `TaskSetDefinedEvent` — фиксирует количество задач

`State_index.yaml` обновляется автоматически. YAML не трогаешь руками.

Проверь результат:

```bash
sdd show-state
```

Должно показать: `phase.current=17`, `phase.status=ACTIVE`, `tasks.total=15`, `tasks.completed=0`.

---

### Шаг 6 — LLM реализует задачи одну за одной

Для каждой задачи говоришь:

```
Implement T-1701
```

LLM:
1. Запускает guard-проверки (`sdd phase-guard`, `sdd task-guard`, `sdd norm-guard`)
2. Читает только файлы из `Inputs:` задачи
3. Пишет только в файлы из `Outputs:` задачи
4. Запускает `sdd complete T-1701` — задача становится DONE

Затем:

```
Validate T-1701
```

LLM:
1. Запускает тесты и проверку инвариантов
2. Создаёт `.sdd/reports/ValidationReport_T-1701.md`
3. Запускает `sdd validate T-1701`

---

### Шаг 7 — Ты наблюдаешь и ревьюишь

В любой момент смотришь состояние:

```bash
sdd show-state
```

Смотришь задачи:

```bash
sdd show-task T-1703 --phase 17
```

Смотришь события:

```bash
sdd query-events --phase 17
```

Если что-то пошло не так — задача реализована неверно, тесты красные и т.п.:

```bash
sdd report-error --type Inconsistency --message "T-1703: тест X не проходит"
```

Это логирует инцидент. Затем говоришь LLM разобраться.

Если обнаружился рассинхрон между State и реальностью:

```bash
sdd sync-state --phase 17
```

Это безопасная идемпотентная команда — пересобирает `State_index.yaml` из EventLog.

---

### Шаг 8 — LLM проверяет Definition of Done

После того как все задачи реализованы и валидированы:

```
Check DoD Phase 17
```

LLM запускает `sdd check-dod --phase 17`. Команда проверяет:

- Все задачи `DONE` (`tasks.completed == tasks.total`)
- `invariants.status == PASS`
- `tests.status == PASS`

Если всё прошло — эмитируется `PhaseCompletedEvent`, статус становится `COMPLETE`.
Если нет — LLM показывает что именно не прошло.

---

### Шаг 9 — LLM пишет отчёт и снапшот

```
Summarize Phase 17
```

LLM создаёт `.sdd/reports/Phase17_Summary.md`.

```
Metrics Report Phase 17
```

LLM создаёт `.sdd/reports/Metrics_Phase17.md` с аномалиями и трендами.

```
EventLog Snapshot Phase 17
```

LLM запускает:

```bash
sdd query-events --phase 17 --include-bash --json --save
```

Создаётся `.sdd/reports/EL_Phase17_events.json` — замороженный аудит-трейл фазы.

---

### Шаг 10 — Ты даёшь финальное добро

Просматриваешь:

- `.sdd/reports/Phase17_Summary.md` — итоги, покрытие, решения
- `.sdd/reports/Metrics_Phase17.md` — аномалии, гипотезы улучшений
- `sdd show-state` — всё COMPLETE

Если доволен — говоришь LLM что фаза принята. Если хочешь правок — указываешь что именно.

После принятия обновляешь `Phases_index.md`:

```markdown
| 17 | Название | .sdd/specs/Spec_v17_*.md | COMPLETE |
```

## Быстрая шпаргалка: команды человека

| Когда | Команда / Действие |
|-------|-------------------|
| Утвердить спецификацию | `cp .sdd/specs_draft/Spec_vN_*.md .sdd/specs/` |
| Активировать фазу | `sdd activate-phase N --tasks T` |
| Посмотреть состояние | `sdd show-state` |
| Посмотреть задачу | `sdd show-task T-NNN --phase N` |
| Посмотреть события фазы | `sdd query-events --phase N` |
| Выровнять State после сбоя | `sdd sync-state --phase N` |
| Залогировать инцидент | `sdd report-error --type T --message "описание"` |
| Отметить фазу COMPLETE в индексе | Редактировать `.sdd/plans/Phases_index.md` вручную |

## Протокол тестирования: агент vs человек

В SDD тесты запускаются в двух режимах — **task mode** (агент, во время реализации задачи)
и **system mode** (ты, в конце фазы). Важно понимать, кто что запускает и почему.

### Что запускает агент

В `task mode` агент при каждой задаче запускает `sdd validate T-NNN`, который внутри
выполняет только команды из `build.commands`, чей ключ **не начинается с `test`**:

| Команда из `build.commands` | Запускается в task mode? |
|-----------------------------|--------------------------|
| `lint` | Да |
| `typecheck` | Да |
| `test` | **Нет** (ключ начинается с `test`) |
| `test_full` | **Нет** (ключ начинается с `test`) |
| `acceptance` | Да (отдельный путь через `_run_acceptance_check`) |

Это инвариант **I-TASK-MODE-1**: агент никогда не запускает полный тест-сьют на каждой задаче.

### Что запускаешь ты

После завершения всех задач фазы ты запускаешь полный прогон тестов вручную:

```bash
# Полный прогон (system mode): lint + typecheck + test + test_full
sdd validate-invariants --phase N --system

# Или напрямую через pytest:
python3 -m pytest tests/ -q

# Проверить статус покрытия:
python3 -m pytest tests/ --cov=src/sdd --cov-report=term-missing -q
```

Это гарантирует, что весь тест-сьют проходит перед закрытием фазы (шаг 8 — Check DoD).

### System mode: когда и зачем

`--system` включает все команды из `build.commands` без исключений.
Используй его когда:

- Финальная валидация фазы перед `Check DoD`
- Хочешь убедиться что ничего не сломалось после серии задач
- Отлаживаешь интеграционные проблемы

```bash
sdd validate-invariants --phase N --system --task T-NNN
```

### Почему такое разделение

| Аспект | Task mode (агент) | System mode (ты) |
|--------|-------------------|------------------|
| Когда | После каждой задачи | В конце фазы |
| Тесты | Пропускаются | Запускаются все |
| Скорость | Быстро (lint + typecheck) | Медленно (полный сьют) |
| Цель | Не тормозить цикл реализации | Финальная гарантия качества |
| Ответственность | LLM | Человек |

Агент намеренно не запускает тест-сьют на каждой задаче, чтобы не тратить время
на полный прогон при каждом изменении. Ты запускаешь его один раз — когда всё готово.

## Что никогда не делать руками

| Что | Почему |
|-----|--------|
| Редактировать `State_index.yaml` напрямую | Это проекция, не источник истины. Изменения затрутся при следующем `sync-state` |
| Писать в `.sdd/specs/` | Спецификации иммутабельны после утверждения |
| Запускать `Implement T-NNN` за несколько задач сразу | Нарушает R.10 (одна задача = одна команда) |
| Вызывать `rebuild_state(EMERGENCY)` | Break-glass операция; используй `sdd sync-state` вместо этого |
| Менять `phase.status` в YAML | Только через `sdd activate-phase` (PLANNED→ACTIVE) или `sdd check-dod` (ACTIVE→COMPLETE) |

---

## Что делать, если что-то сломалось

### State рассинхронизировался

```bash
sdd sync-state --phase N
sdd show-state  # проверить
```

### Тесты красные после реализации задачи

```bash
sdd show-task T-NNN --phase N  # проверить Outputs и Acceptance
# скажи LLM какой тест и что не так
```

### Фаза не активируется (guard rejection)

```bash
sdd show-state  # проверить что текущая фаза COMPLETE
# только одна фаза может быть ACTIVE
```

### LLM не может найти файл задачи

```bash
sdd show-task T-NNN  # LLM должен пользоваться этим, не читать .sdd/ напрямую
```

### Неизвестная ошибка в JSON stderr

Читаешь поля `stage` и `error_type` — они дают точный FS-* сценарий из `.sdd/docs/ref/recovery.md`.
Передаёшь LLM полный JSON — он классифицирует и применяет нужный RP-*.

## Структура артефактов фазы

```
.sdd/
  specs/
    Spec_v17_*.md          ← утверждённая спецификация (иммутабельна)
  specs_draft/
    Spec_v17_*.md          ← черновик (до утверждения)
  plans/
    Plan_v17.md            ← план реализации
    Phases_index.md        ← реестр всех фаз
  tasks/
    TaskSet_v17.md         ← задачи с Input/Output/Acceptance
  runtime/
    State_index.yaml       ← текущее состояние (только читать)
    audit_log.jsonl        ← fallback-лог при недоступном DuckDB
  reports/
    ValidationReport_T-17xx.md   ← отчёт по каждой задаче
    Phase17_Summary.md           ← итог фазы
    Metrics_Phase17.md           ← метрики и аномалии
    EL_Phase17_events.json       ← снапшот EventLog
  state/
    sdd_events.duckdb      ← единственный источник истины (EventLog)
```
